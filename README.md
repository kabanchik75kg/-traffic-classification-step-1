# Контракт проекта ВКР

## Базовая информация
- **Тема:** Разработка средства классификации сетевого трафика на основе 
  поведенческих признаков с применением методов машинного обучения.
- **Вуз:** МГТУ им. Н.Э. Баумана, специалитет, 6 курс.
- **Задача:** бинарная классификация сетевых потоков (benign / malicious) 
  на поведенческих признаках, без анализа payload и без использования 
  IP-адресов и портов как признаков.

## Роль Claude
Научный руководитель. Аргументирует свою позицию, не подстраивается под 
мнение студента, если оно неубедительно. Соглашается, только когда 
аргументы студента действительно меняют картину. Студент попросил это 
явно — соблюдать всегда.

## Принятые решения

### Датасеты
- **Основной:** LycoS-Unicas-IDS2018 (исправленная CSE-CIC-IDS2018, 
  77 признаков, извлечённых через LycoSTand).
- **Для cross-dataset evaluation:** LycoS-IDS2017 (исправленная CIC-IDS2017 
  от тех же авторов, совместимая схема признаков).
- **Дополнительная проверка:** CTU-13 (реальный botnet-трафик).
- **Опционально, для графовой части:** MCFP + MTA — только при наличии времени.

### Этапы работы
1. Baseline на flow-признаках (LogReg, RandomForest, XGBoost/LightGBM).
2. Sequence-based подход (1D-CNN, BiLSTM по последовательности пакетов 
   и/или burst-структуре).
3. Graph-based подход (GraphSAGE / RGAT на burst-графах) — опционально.

Стратегия: от простого к сложному, чтобы на каждом этапе уже было что 
показывать. Не уходить в графовые модели до получения работающего baseline.

### Методика
- **Разбиение train/test:** по времени (по дням захвата), не случайное.
- **Cross-dataset evaluation обязательно**, не как «если успею».
- **Признаки:** без IP, портов, timestamp. Готовый CSV LycoS-Unicas 
  используется как реализация отображения PCAP → признаки.
- **Метрики:** F1 (macro), PR-AUC, ROC-AUC, FPR при фиксированном 
  TPR=0.95. Accuracy — вспомогательная.
- **Дисбаланс классов:** class weights / focal loss / undersampling. 
  SMOTE на flow-данных без аккуратности не использовать.

### Прототип «средства»
Требование темы ВКР («Разработка средства...»). 
CLI/сервис: NFStream → flow-признаки → сохранённая модель → классификация. 
Делается параллельно с экспериментами, не в самом конце.

## Технический стек (по умолчанию)
- Python 3.11+, pandas, scikit-learn, xgboost, lightgbm, pytorch, 
  pytorch-geometric (для GNN), NFStream.
- Эксперименты — Jupyter / отдельные .py-скрипты, версионирование через git.



# Лог решений по проекту

Этот документ объясняет, ПОЧЕМУ приняты решения, перечисленные в 
PROJECT_CONTRACT.md. Используется, когда в новом диалоге возникает 
вопрос «а почему мы выбрали X, а не Y».

## 1. Почему LycoS-Unicas-IDS2018, а не CIC-IDS2017

CIC-IDS2017 и CSE-CIC-IDS2018 имеют документированные дефекты:
- Ошибки в реализации DoS Hulk.
- Некорректная обработка TCP в flow construction (CICFlowMeter).
- Ошибки в feature extraction в самом CICFlowMeter.
- Дублирование пакетов, packet misorder, некорректная разметка 
  части атак (Lanvin et al., Engelen et al.).

LycoS-Unicas-IDS2018 — это CSE-CIC-IDS2018, перегенерированный через 
LycoSTand (Rosay et al., 2022) с исправлением вышеперечисленного. 
77 признаков, корректная обработка TCP-сессий.

## 2. Почему LycoS-IDS2017 как второй датасет, а не BCCC-CSE-CIC-IDS2018

Для cross-dataset evaluation нужны:
- Совместимая схема признаков (LycoSTand в обоих случаях).
- Разные сетевые окружения (2017 — физический testbed, 2018 — AWS).

BCCC имеет другое признаковое пространство — пришлось бы городить 
маппинг. LycoS-IDS2017 решает проблему «из коробки».

Дополнительно: известно, что модели, обученные на CSE-CIC-IDS2018, 
дают почти идеальную accuracy внутри датасета, но при тесте на 
LycoS-IDS-2017 показывают результаты около случайных. Это значит, 
что внутри-датасетные 99%+ не отражают реальную работоспособность. 
Cross-dataset evaluation — главный честный тест.

## 3. Почему отказ от IP/портов/timestamp

Стандартная ошибка работ по CIC-датасетам: модель учит 
«атаки идут с этого IP» или «portscan на этом порту», а не 
поведенческие паттерны. Это нарушает смысл темы 
(«поведенческие признаки»). Без IP/портов модель вынуждена учиться 
на статистике потока — это и есть поведение.

## 4. Почему этапы flow → sequence → graph, а не сразу graph

- Flow-baseline (RF, XGBoost) на 77 числовых признаках 
  гарантированно даёт результат за день работы. Это страховка: 
  даже если всё остальное провалится, есть что защищать.
- Sequence-based подход (1D-CNN, BiLSTM по N первым пакетам) 
  захватывает временную структуру, которую теряет агрегированная 
  статистика. Это «средняя» по сложности часть.
- Graph-based (burst-граф + GraphSAGE/RGAT) — текущий SOTA, 
  но требует много инженерной работы и риск не успеть. 
  Берём в последнюю очередь.

## 5. Почему time-split, а не случайный

Случайное разбиение flow-данных протекает: соседние потоки атаки 
попадают в train и test одновременно, модель угадывает по сторонним 
корреляциям. В LycoS-Unicas есть метка времени / разбиение по дням — 
честный сплит делается по ним.

## 6. Почему F1 / PR-AUC / FPR@TPR, а не accuracy

Дисбаланс классов в LycoS-Unicas: benign ~73%, остальное — атаки. 
Accuracy 95% получается тривиально (модель всегда предсказывает benign).
FPR при фиксированном TPR — стандартная IDS-метрика: показывает, 
сколько ложных тревог на единицу обнаружения. На реальных объёмах 
трафика 0.1% FPR — это тысячи ложных алертов в час.

## 7. Почему RandomForest как основной baseline, а не только LogReg

LogReg оставляем как нижнюю границу здравомыслия (если RF не сильно 
лучше LogReg — значит, признаки тривиальные). RF / XGBoost — 
практический baseline, они обычно дают сильный результат на 
табличных flow-данных и часто бьют DL.

## 8. По матпостановке (из обсуждения):

- Выходы RF/GB не являются откалиброванными вероятностями — 
  это надо явно оговорить в формальной постановке. При 
  необходимости — Platt scaling или isotonic regression.
- Признаковое пространство привязывается к LycoSTand как 
  конкретной реализации отображения φ: PCAP → R^d.
- Стандартизация — только для моделей, чувствительных к масштабу 
  (LogReg, NN), не для деревьев.
- Категориальный proto разворачивается в q компонент one-hot 
  в итоговом векторе x_i (формула должна это отражать).

## 9. Что НЕ берём и почему

- IoT-датасеты (TON_IoT, CIC-IoT-2023): расфокусирует тему.
- Подходы «PCAP → grayscale image → CNN»: это не поведенческая 
  классификация по смыслу, а распознавание байтовых паттернов 
  (ближе к сигнатурам). Не соответствует теме.
- Гонка за accuracy на одном датасете: на CIC-данных у всех 99%, 
  дальнейший рост — шум. Не результат для ВКР.
- SMOTE на flow-данных без аккуратности: генерирует нереалистичные 
  потоки в промежутках между классами.
